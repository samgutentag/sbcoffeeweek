# Improvements to Backport to sbburgerweek

Things built or fixed during the Coffee Week fork that should go back to the burger week repo for next year's event.

## 1. `fetch-details.py` — Auto-populate website and phone from Google Places API

**What:** New script that reads `place-ids.json`, calls the Places Details API for `website` and `formatted_phone_number` fields, and updates both `data-YYYY.js` and `data.js` in place. Only overwrites `null` values — existing data is preserved.

**How to port:** Copy `fetch-details.py` to the burger week repo. No other changes needed — it reads from `place-ids.json` (which already exists there) and uses the same `GOOGLE_PLACES_API_KEY` env var.

**Usage:**
```bash
GOOGLE_PLACES_API_KEY=xxx python3 fetch-details.py
```

**Add to README:** Under Step 11 (Restaurant Hours), add a note about this script as a companion to `fetch-place-ids.py` and `fetch-hours.py`.

## 2. Stats footer date — Read from config instead of hardcoded

**What:** `stats/stats.js` had `"Feb 19"` hardcoded in the footer note. Now reads from `THEME.eventStartDate` and formats it dynamically.

**File:** `stats/stats.js`

**Change:** Replace the hardcoded string:
```js
// Before (hardcoded)
note.textContent =
  "Updated every 5 minutes. Data collected since event start (Feb 19).";

// After (config-driven)
var startLabel = THEME.eventStartDate
  ? new Date(THEME.eventStartDate + "T00:00:00").toLocaleDateString("en-US", { month: "short", day: "numeric" })
  : "event start";
note.textContent =
  "Updated every 5 minutes. Data collected since event start (" + startLabel + ").";
```

## 3. `tracking-snapshot.js` cleanup

**What:** When forking the repo, `tracking-snapshot.js` (the pre-baked snapshot for off-season stats display) carries over with stale data from the previous event. It needs to be deleted in the fork — otherwise the stats page shows old event data.

**Recommendation:** Add a note to the README's fork guide (Step 5 or the Event Day Checklist) reminding forkers to delete `tracking-snapshot.js` if it exists. It'll be regenerated by the snapshot workflow during the new event.

Also consider adding it to `.gitignore` or having `apply-theme.py` clear it when `trackUrl` changes.

## 4. Google Maps search URL fallback pattern

**What:** For locations without a `maps.app.goo.gl` short link, the long-form Google Maps search URL works as a `mapUrl` value:
```
https://www.google.com/maps/search/?api=1&query=Business+Name+123+Main+St+City+CA
```

`fetch-place-ids.py` resolves these to Place IDs correctly, and the directions buttons handle them. This is a useful pattern for quickly populating data before manually collecting short links.

**Recommendation:** Document this in the README's Step 4 (Add Your Restaurant Data) as a quick-start alternative to manually copying share links from Google Maps.

## 5. `python3 -m http.server` binding fix

**What:** On newer macOS with Python 3.14+, `python3 -m http.server 8000` fails with a `socket.gaierror`. The fix is to bind explicitly:
```bash
python3 -m http.server 8000 --bind 127.0.0.1
```

**Recommendation:** Update the README's "Run Locally" section and Step 1 to use `--bind 127.0.0.1`.

---

## 6. Dataset name in SQL queries

**What:** `apply-theme.py` updates `toDateTime` timestamps in `workers/track/index.js` and `snapshot-tracking.yml`, but does NOT update the Analytics Engine dataset name in SQL `FROM` clauses. When forking, these must be manually changed from `sbburgerweek` to the new dataset name.

**Options:**
- (A) Add dataset name replacement to `apply-theme.py` (derive from `storageKey` or add a new config field)
- (B) Document it clearly in the fork guide (currently not mentioned)

Recommendation: Option A is cleaner long-term. Add a `datasetName` field to `config.js` and have `apply-theme.py` replace `FROM <old>` with `FROM <new>` in both files.
